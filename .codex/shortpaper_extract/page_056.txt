 
4.4  Database  Design  
This  database  is  the  backbone  of  the  ePustaka  Munshi  system  because,  prior  to  its  
implementation,
 
there
 
was
 
a
 
physical
 
logbook
 
used
 
in
 
the
 
school,
 
which
 
was
 
very
 
fragile
 
and
 
easily
 
damaged.
 
The
 
database
 
model
 
used
 
is
 
a
 
relational
 
database
 
model
 
running
 
on
 
PostgreSQL
 
through
 
Supabase.
 
With  regard  to  domain  analysis  and  the  process  of  normalization,  the  database  
schema
 
will
 
comprise
 
the
 
following
 
important
 
entities:
 
1.  Roles:  To  satisfy  the  Role-Based  Access  Control  requirement,  the  following  
table
 
specifies
 
the
 
order
 
of
 
the
 
user
 
role
 
(for
 
instance,
 
Librarian,
 
Library
 
Prefect,
 
and
 
Student)
 
and
 
does
 
not
 
mix
 
role
 
definitions.
 2.  Users:  This  table  holds  the  profile  and  login  details  of  all  participants  in  the  
system.
 
It
 
directly
 
relates
 
to
 
the
 
"Roles"
 
table
 
for
 
assigning
 
rights
 
and
 
contains
 
primary
 
information
 
such
 
as
 
"Student_ID_Number"
 
and
 
"Status."
 3.  Categories:  This  entity  helps  to  systematize  the  categories  of  books  
classified
 
in
 
a
 
library
 
using
 
Dewey
 
Decimal
 
Classification.
 
This
 
will
 
avoid
 
the
 
duplication
 
of
 
categories
 
such
 
as
 
“Sci-Fi”
 
and
 
“Science
 
Fiction.”
 4.  Books:  The  primary  database  for  holding  information  for  the  library.  It  
maintains
 
all
 
the
 
bibliographic
 
information
 
like
 
Title,
 
Author,
 
Publisher,
 
ISBN,
 
Year
 
Of
 
Publication,
 
and
 
importantly,
 
the
 
Status
 
information
 
like
 
“Available,
 
Borrowed,
 
Lost,
 
Damaged”.
 5.  Transactions:  This  is  intended  to  replace  the  borrowing  logbook.  All  
circulation
 
events
 
are
 
recorded
 
by
 
this
 
entity.
 
This
 
can
 
be
 
seen
 
as
 
the
 
meeting
 
point
 
between
 
the
 
User
 
(Student)
 
and
 
the
 
Book.
 
The
 
Borrow_Date
 
is
 
recorded,
 
the
 
Due_Date
 
is
 
calculated,
 
and
 
the
 
Return_Date
 
is
 
marked.
 6.  Fines:  This  is  responsible  for  the  business  rule  related  to  overdue  materials.  It  
connects
 
the
 
Transactions
 
related
 
to
 
the
 
calculation/reporting
 
status
 
of
 
the
 
fee
 
for
 
late
 
returns.
 7.  OCR_Logs:  Unique  to  this  project,  this  table  serves  as  a  staging  area  for  the  
purpose
 
of
 
the
 
digitization
 
process.
 
The
 
raw
 
texts
 
obtained
 
from
 
the
 
ledgers
 
that
 
users
 
have
 
uploaded,
 
alongside
 
the
 
URL
 
of
 
the
 
image
 
and
 
its
 
verification
 
status,
 
are
 
stored
 
in
 
this
 
table
 
in
 
order
 
to
 
ensure
 
cleanliness
 
of
 
the
 
data
 
before
